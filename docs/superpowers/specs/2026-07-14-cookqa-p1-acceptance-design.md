# CookQA P1 真实验收收口设计

## 目标

在现有 200 道菜真实索引上，统一构建阶段与运行时的食材别名规则，修正“辣/不辣”等主观表达的硬条件语义，并输出可重复的预热 P50/P95 评测结果。只有所有目标达标后，README 才声明 P1 完成。

## 设计

- `build_runtime()` 读取 `config/ingredient_aliases.json`，将别名映射传入 `QueryRouter`；构建与运行时使用同一份规范化表。
- `QueryRouter` 将主观词作为条件标签而不是食材名：`不辣/无辣` 转换为结构化排除标签 `spicy`，`辣/香辣` 转换为结构化要求标签 `spicy`。保留用户明确写出的真实食材排除规则。
- `satisfies_hard_filters()` 通过可配置的结构化标签判断辣味，优先使用菜谱食材名称和规范化别名；无法从结构化字段确认时不得宣称硬条件已满足。
- benchmark 在真实 API 上执行固定 50 条查询，多轮预热后记录搜索与流式回答首字延迟的 P50/P95，并将达标条件写入 `Data/runtime/benchmark-report.json`。
- README 和 `docs/UNFINISHED.md` 只引用当前机器实测结果；若任一指标不达标，继续优化检索/过滤/服务路径后再更新完成状态。

## 验收标准

- 运行时别名查询与构建阶段一致。
- “推荐不辣的虾菜”不得返回结构化确认的辣味菜；“推荐辣味菜”不得返回无辣证据的菜。
- 固定评测集全部成功，Recall@5 >= 0.90，可靠硬条件违规数为 0。
- 预热搜索 P95 <= 1000ms，详细回答首字 P95 <= 3000ms。
- 报告包含 P50/P95、样本数、失败数、硬条件违规数和明确的 targets。
