# CookQA 本地 Graph RAG MVP 设计

- 日期：2026-07-13
- 状态：已由用户确认
- 目标平台：Windows 本机
- 数据源：[Anduin2017/HowToCook](https://github.com/Anduin2017/HowToCook)

## 1. 背景与目标

CookQA 是一个面向中文菜谱查询的本地 Graph RAG 系统。MVP 从 HowToCook 中选择约 200 道菜，使用 Neo4j 表达菜谱、食材、分类、烹饪方法和工具之间的关系，使用 BM25 与 FAISS 分别完成稀疏检索和稠密检索，并通过 FastAPI 与轻量 Web UI 对外提供查询能力。

系统的首要目标是正确、迅速地返回用户想要的菜谱。推荐列表与自然语言生成分离：推荐结果不等待 LLM，用户选中菜谱后再查看完整详情或请求流式说明。

## 2. 已确认范围

### 2.1 MVP 能力

系统支持以下查询类型：

1. 精确菜谱查询，例如“宫保鸡丁怎么做”。
2. 食材反查，例如“鸡蛋和番茄能做什么”。
3. 条件推荐，例如“推荐 20 分钟内、不辣的鸡肉菜”。
4. 语义推荐，例如“想吃清淡又下饭的菜”。
5. 相似菜谱，例如“找和鱼香肉丝相似但不含猪肉的菜”。
6. 菜谱比较，例如“宫保鸡丁和辣子鸡有什么区别”。

### 2.2 MVP 不做

- 步骤级向量索引或全库步骤检索。
- 在线增量导入和热更新。
- 用户账号、权限和多租户。
- 公网部署。
- 多模态图片检索。
- 医疗或营养建议。
- Agent 自动执行菜谱操作。
- 云端模型或云端数据库。

对于“鸡蛋什么时候下锅”这类缺少具体菜谱的操作问题，系统要求用户补充菜名，不把多个菜谱的步骤混合成一个答案。

## 3. 非功能要求

### 3.1 本地与安全

- 全部组件在 Windows 本机运行，不使用 Docker。
- LLM 与 Embedding 使用本机 Ollama。
- 默认聊天模型为 `qwen3.5:4b`。
- 默认 Embedding 模型为 `bge-m3`。
- Neo4j 使用 Windows ZIP 原生发行版，开发阶段控制台运行，稳定后可安装为 Windows 服务。
- 服务默认只绑定 `127.0.0.1`。
- Neo4j 密码等敏感配置只从本地环境变量读取，不写入源码、索引清单或日志。

### 3.2 性能目标

- 预热状态下，推荐列表响应时间 P95 不超过 1 秒。
- 预热状态下，详细回答首字时间 P95 不超过 3 秒。
- 冷启动耗时单独统计，不混入预热指标。
- 性能目标必须通过当前机器上的基准测试验证，不能仅根据架构推测宣布达标。

## 4. 总体架构

系统由以下模块组成：

### 4.1 Web UI

使用静态 HTML、CSS 和 JavaScript，由 FastAPI 直接托管。Web UI 负责：

- 接收自然语言查询。
- 展示 Top 5 菜谱卡片。
- 展示匹配原因、来源、推断字段和降级状态。
- 在用户选择菜谱后展示完整结构化详情。
- 按需展示 `qwen3.5:4b` 的流式回答。

### 4.2 FastAPI 接口层

接口层负责参数校验、调用查询服务、返回结构化结果、管理流式输出以及隐藏 Neo4j、FAISS、BM25 和 Ollama 的内部实现。

### 4.3 查询路由器

MVP 使用规则、菜名/食材词典和查询特征生成查询计划。LLM 不参与检索路由，不生成 Cypher，不决定检索权重。

### 4.4 检索协调器

协调器按查询计划调用 BM25、FAISS 和 Neo4j，执行硬过滤、RRF 融合、降级处理和结果解释。

### 4.5 生成器

生成器只在菜谱已经确定后调用 `qwen3.5:4b`，并且只根据结构化菜谱详情生成回答。生成失败不影响菜谱卡片和原始详情。

### 4.6 数据构建器

数据构建器将 HowToCook Markdown 转换为统一的 `recipes.jsonl`，再从同一份中间数据构建 Neo4j、FAISS 和 BM25。

## 5. 数据范围与数据模型

### 5.1 MVP 数据选择

默认选择约 200 道菜，覆盖素菜、荤菜、水产、早餐、主食、汤与粥等主要分类。饮料、调酒、单独酱料和半成品加工暂不纳入 MVP。

选中的源文件路径保存在固定清单中，并记录 HowToCook Git 提交版本，确保数据集可复现。

### 5.2 规范化菜谱

每道菜的统一记录至少包含：

- 稳定的 `recipe_id`。
- 菜名、别名、分类和简介。
- 原料、用量、单位、是否可选以及原始写法。
- 难度、卡路里和明确耗时。
- 烹饪方法和工具。
- 完整操作步骤。
- 标签以及标签的来源和可信度。
- 原始文件相对路径和源数据版本。

`recipe_id` 根据源文件相对路径稳定生成。食材规范名称与原始写法同时保存，例如“番茄”和“西红柿”映射到同一个规范实体。

### 5.3 事实与推断

- 原文明确提供的难度、卡路里、用量和耗时标记为原始事实。
- 辣、少油、快手等由规则推断的标签必须记录推断依据。
- 系统不使用 LLM 自动补全缺失的营养、耗时或食材信息。
- 用户提出强制条件时，字段缺失视为无法确认，不能冒充满足条件。

### 5.4 Neo4j 图模型

主要节点：

- `Recipe`
- `Ingredient`
- `Category`
- `Method`
- `Tool`
- `Tag`

主要关系：

- `Recipe -[:REQUIRES]-> Ingredient`
- `Recipe -[:BELONGS_TO]-> Category`
- `Recipe -[:USES_METHOD]-> Method`
- `Recipe -[:USES_TOOL]-> Tool`
- `Recipe -[:HAS_TAG]-> Tag`

`REQUIRES` 关系保存用量、单位、是否可选和原始描述。操作步骤不建立独立图节点，也不建立步骤级索引；完整步骤作为菜谱详情保存。

系统不预先存储所有菜谱两两之间的相似关系。相似性由共同食材、烹饪方法和向量检索动态计算，避免冗余边。

## 6. 稀疏索引与稠密索引

### 6.1 BM25

BM25 负责字面与关键词召回，重点覆盖：

- 菜名和别名。
- 食材。
- 分类。
- 烹饪方法和工具。
- 简介和标签。
- 低权重的操作步骤文本。

BM25 实际持久化倒排索引、词频、文档长度和 IDF 等统计信息。菜名权重最高，食材和别名次之。

### 6.2 FAISS

FAISS 负责菜谱级语义召回。`bge-m3` 将每道菜的检索摘要转换为稠密向量，FAISS 保存向量以及与 `recipe_id` 的映射。

MVP 数据量约 200 道菜，使用精确相似度搜索，不引入近似索引参数。

### 6.3 ID 与版本一致性

BM25、FAISS、Neo4j 和完整菜谱详情使用同一套 `recipe_id`。每次构建生成版本清单，记录数据版本、菜谱数量、ID 哈希、Embedding 模型、向量维度和各索引版本。

## 7. 查询路由

路由器输出：

- `intent`
- `recognized_recipes`
- `required_ingredients`
- `excluded_ingredients`
- `constraints`
- `retrieval_strategy`
- `confidence`

路由策略：

| 查询类型 | 策略 |
|---|---|
| 精确菜谱 | 菜名/别名直接匹配 |
| 食材反查 | Neo4j 食材关系 + BM25 + FAISS |
| 条件推荐 | Neo4j 硬过滤 + 混合排序 |
| 语义推荐 | FAISS 为主，BM25 补充 |
| 相似菜谱 | FAISS 相似度 + Neo4j 共同关系 |
| 菜谱比较 | Neo4j 关系比较 |
| 缺少必要上下文 | 请求用户补充信息 |

低置信度的普通推荐进入完整混合检索；缺少菜名的操作步骤问题要求澄清。

## 8. 混合检索与硬过滤

### 8.1 执行流程

1. 规范化查询并生成查询计划。
2. 精确菜名或别名命中时直接返回。
3. 需要推荐时并行执行 BM25、FAISS 和 Neo4j。
4. 各路返回候选 `recipe_id` 排名。
5. Neo4j 对强制条件进行硬过滤。
6. 使用 RRF 根据名次融合保留下来的结果。
7. 返回 Top 5、匹配原因、来源、检索策略和耗时。

### 8.2 硬过滤

硬过滤适用于明确的必须包含/排除食材、分类、最大耗时、工具和可靠的难度范围。不满足条件的菜谱直接移除，而不是仅降低排名。

“清淡”“下饭”“适合夏天”等主观表达只作为软排序因素，除非后续有可靠、可验证的结构化数据支持。

### 8.3 RRF

BM25、FAISS 和图查询的原始分数不可直接相加。系统使用 RRF 按各自名次融合。初始版本采用固定、可解释的权重，后续根据固定评测集调优，不由 LLM 临时决定。

## 9. 可降级执行

正常路径为 Neo4j + BM25 + FAISS。

| 故障 | 处理 |
|---|---|
| `bge-m3` 或 FAISS 超时 | 使用 BM25 + Neo4j |
| BM25 不可用 | 使用 FAISS + Neo4j |
| Neo4j 不可用且无硬条件 | 使用 BM25 + FAISS |
| Neo4j 不可用且存在硬条件 | 不声称结果满足条件，返回明确提示 |
| `qwen3.5:4b` 不可用 | 返回菜谱卡片和原始详情，不生成回答 |
| 所有检索索引不可用 | 返回服务错误，不让模型凭空回答 |

降级信息在 API 和 Web UI 中显式展示，包括不可用组件和实际检索策略。降级可以减少召回能力，但不能静默牺牲明确条件的正确性。

## 10. 性能策略

预热状态下的内部时间预算：

| 阶段 | 预算 |
|---|---:|
| 查询规范化与路由 | 50 ms |
| BM25 | 50 ms |
| Neo4j 查询 | 150 ms |
| `bge-m3` 向量化 + FAISS | 700 ms |
| 过滤、RRF 与序列化 | 50 ms |

这些阶段尽量并行，预算不是简单相加。具体措施包括：

- 启动时加载 BM25 和 FAISS 到内存。
- 复用 Neo4j 连接池。
- 保持 `bge-m3` 预热。
- BM25、Neo4j 和查询向量计算并行执行。
- 对规范化后的重复查询使用短期内存缓存。
- 精确菜名查询跳过 Embedding。
- 稠密检索超过预算时及时降级。
- 推荐结果返回后再执行 LLM 生成。
- 记录各阶段耗时与降级状态，不记录敏感配置。

## 11. 数据导入与更新

### 11.1 本地目录

```text
Data/
├── source/howtocook/
├── processed/recipes.jsonl
├── indexes/
├── neo4j/
└── runtime/
```

运行数据不提交到 Git。仓库只保留选择清单、解析规则、示例配置和构建代码。

### 11.2 构建流程

1. 验证选择清单中的源文件全部存在。
2. 使用确定性 Markdown 解析器生成 `recipes.jsonl`。
3. 检查字段、ID、重复项和关系完整性。
4. 使用 `bge-m3` 构建菜谱级 FAISS。
5. 构建菜谱级 BM25。
6. 导入 Neo4j 节点与关系。
7. 执行跨索引一致性检查。
8. 生成版本清单。
9. 全部验证通过后才允许 FastAPI 加载。

选中的菜谱解析失败时，本次构建失败并生成报告，不静默跳过。

### 11.3 更新与回滚

MVP 不做在线热更新。更新时停止 API 或进入维护状态，在临时目录完整构建新版本，验证后再切换。保留上一版索引和 Neo4j 备份，构建失败时继续使用上一版。

FastAPI 启动时校验三套索引的版本、菜谱数量、ID 哈希、模型和向量维度。一致性失败时 `/ready` 返回未就绪，不使用混合版本提供查询。

## 12. API 设计

### 12.1 搜索

`POST /api/v1/search`

返回 Top 5 菜谱、匹配原因、检索策略、阶段耗时、来源和降级状态。

### 12.2 菜谱详情

`GET /api/v1/recipes/{recipe_id}`

直接返回完整结构化菜谱，不调用 LLM。

### 12.3 流式回答

`POST /api/v1/recipes/{recipe_id}/answer/stream`

针对已经确定的菜谱和可选的用户问题流式生成说明。用户取消请求时终止对应的 Ollama 流。

### 12.4 健康检查

- `GET /health`：检查 FastAPI 进程是否存活。
- `GET /ready`：检查 Neo4j、索引版本和 Ollama 是否可以提供完整服务。

搜索与生成使用独立接口，保证推荐列表不被 LLM 阻塞。

## 13. Web UI 设计

页面提供查询输入框、示例问题、加载状态和 Top 5 菜谱卡片。卡片展示：

- 菜名、分类、难度和耗时。
- 主要食材。
- 匹配原因。
- 原始字段和推断字段提示。
- HowToCook 来源。
- 降级状态。

精确菜名查询可自动打开唯一结果；推荐查询等待用户选择。用户选中菜谱后立即展示结构化详情，再按需启动流式自然语言说明。

## 14. 错误处理

- 查询为空或格式错误时返回参数错误。
- 菜谱不存在时返回 404，不让 LLM猜测。
- 索引版本不一致时服务进入未就绪状态。
- 硬条件无法验证时明确说明，不返回误导结论。
- Ollama 不可用时保留结构化菜谱体验。
- 前端不显示堆栈信息。
- 日志不记录密码、Token、Cookie、Authorization 请求头或其他敏感信息。

## 16. 验收标准

### 16.1 功能

- 成功导入约 200 道菜。
- 六类查询均有可验证的行为。
- Top 5 卡片包含匹配依据和来源。
- 完整菜谱无需 LLM 即可查看。
- 全程不需要 Docker 或云端 API。

### 16.2 质量

- 菜名及别名查询 Top 1 准确率为 100%。
- 固定评测集期望菜谱 Recall@5 不低于 90%。
- 对具有可靠数据的硬条件，违规结果为 0。
- 无法验证的条件明确标记，不冒充满足。

### 16.3 性能

- 预热状态推荐列表 P95 不超过 1 秒。
- 预热状态详细回答首字 P95 不超过 3 秒。
- 冷启动单独记录。
- 性能未达标时必须报告具体瓶颈和测量结果。

## 17. 关键设计结论

1. 使用无 Docker 的本地部署方案。
2. 使用 Neo4j + FAISS + BM25 + Ollama + FastAPI + 静态 Web UI。
3. 只建立菜谱级检索索引，不建立步骤级索引。
4. 查询路由采用确定性规则，LLM 不在一秒检索关键路径中。
5. Neo4j 负责关系查询和硬条件验证，BM25 与 FAISS 负责稀疏和稠密召回。
6. 使用 RRF 融合排名，使用明确、可观测且不牺牲正确性的降级策略。
7. 推荐列表和自然语言生成分离。
8. 三套索引只从同一份规范化数据构建，并通过版本清单强制一致。
