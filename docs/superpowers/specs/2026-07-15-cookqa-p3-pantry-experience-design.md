# CookQA Phase 3 食材驱动产品体验设计

## 状态

- 日期：2026-07-15
- 产品方向：已确认
- 实现状态：未开始
- 主线：B，按现有食材找菜
- 轻量增强：C，当前菜谱单轮问答

## 目标

Phase 3 将 CookQA 从可演示的搜索页面升级为可完成“盘点现有食材、选择菜谱、查看缺料、
进入下厨指导”流程的本地菜谱应用。当前固定 200 道菜、检索架构和 Phase 2 性能门槛保持
不变，本阶段优先补齐产品功能与浏览器体验。

核心用户流程为：

```text
输入现有食材
  -> 增加排除食材、最长耗时和口味条件
  -> 查看可以直接做 / 缺 1-2 种主要食材 / 其他相关建议
  -> 打开菜谱并核对已有、缺少和可选食材
  -> 查看步骤
  -> 针对当前菜谱发起一次轻量问答
```

现有自由文本搜索继续保留。食材模式是并列入口，不使用拼接自然语言的方式模拟结构化
请求，也不改变现有六类查询的路由和检索语义。

## 非目标

Phase 3 不扩大当前 200 道菜数据集，不修改 BM25、FAISS、Neo4j 或 RRF 检索架构和权重。
本阶段也不实现多轮对话、跨菜谱记忆、自主 Agent、用户账号、云同步、购物车，以及缺乏
可靠数据支撑的营养、热量或份量换算。

## 产品范围

### 双入口首页

Web UI 保留“智能搜索”，并增加“按现有食材找菜”。两种模式共享结果区和菜谱详情，但
请求和输入状态独立：

- 智能搜索继续调用 `POST /api/v1/search`；
- 食材模式调用新的结构化 `POST /api/v1/pantry/search`；
- 切换模式不自动提交，也不把结构化条件改写成自然语言；
- Phase 3 不持久化未提交输入或个人状态。

### 食材输入

食材模式提供已有食材、排除食材、最长耗时、不辣和默认基础调料开关。已有食材至少包含
一项；排除食材和可靠的耗时、不辣约束均作为硬条件。

默认基础调料集合固定放在 `config/pantry_staples.json`，复用现有食材规范化和别名规则。
该集合
默认开启、界面可见且可关闭，只影响缺料计算，不从菜谱详情中隐藏真实食材。用户明确排除
某项时，排除条件优先。首版不提供基础调料编辑器，避免扩展成库存管理系统。

### 匹配与分组

匹配基于规范化后的非可选食材：

1. `ready`：除默认基础调料外，没有缺少的必需食材；
2. `near`：缺少 1–2 种必需食材；
3. `related`：缺少超过 2 种，但已有食材覆盖率不低于 50%。

可选食材不计入缺失。命中排除食材的菜谱不得进入任何分组；设置最长耗时后，耗时未知的
菜谱不能被当作满足条件返回。

每组最多返回 5 道菜。组内依次按缺少食材数升序、覆盖率降序、命中食材数降序和菜谱名称
升序，保证重复请求顺序稳定。固定 200 道菜规模下直接遍历当前活动版本的规范化菜谱，不
增加新索引，也不调用 LLM。

### 结果与详情

食材结果卡片展示分组、覆盖率、已有食材、缺少食材、可选食材、耗时、难度和匹配原因。
详情继续展示完整来源、食材和步骤，并区分已有、缺少、基础调料和可选食材。不得为改善
显示而伪造缺失的耗时、难度、份量或营养信息。

### 当前菜谱轻量问答

轻量问答复用现有 `POST /api/v1/recipes/{recipe_id}/answer/stream`。每次请求只携带当前
菜谱和本次问题，不保存或发送历史回答。界面提供四个快捷问题：

- 缺少某种食材时可以怎么替代；
- 哪一步最容易失败；
- 没有某个厨具时怎么办；
- 概括制作顺序。

用户仍可输入自定义问题。界面支持停止、重试和复制；新的请求替换上一条展示结果。Ollama
不可用时只影响问答区域，食材匹配、自由搜索和菜谱详情继续可用。

## 架构与组件边界

### API 模型

新增 `PantrySearchRequest`、`PantryMatch` 和 `PantrySearchResponse`。请求包含已有食材、
排除食材、最长耗时、不辣和是否使用默认基础调料；响应包含规范化输入、三组结果、警告和
耗时。已有和排除食材列表分别最多 30 项，单项去除首尾空白后为 1–50 个字符，最长耗时
范围为 1–1440 分钟。规范化后没有有效已有食材时返回 422；同一食材同时出现在已有和排除
列表时返回 422，不自行猜测用户意图。

### 匹配服务

食材匹配实现为独立领域组件。它只依赖当前活动版本的 `Recipe` 集合、现有别名规范化器和
基础调料配置，不依赖 FastAPI、Neo4j、FAISS 或 Ollama。路由只负责校验和映射，不承载排序
规则；Web 层只渲染响应，不自行重复计算分组。

### Web UI

继续使用静态 HTML、CSS 和原生 JavaScript，不引入前端框架。新增代码按输入状态、请求、
结果渲染和问答流控制分责，不重构无关页面。模式切换、食材标签、结果卡和问答控制必须
支持键盘操作和可见焦点，并在约 390px 窄屏和桌面宽度下可用。

## 数据流

```text
浏览器结构化输入
  -> Pydantic 请求校验
  -> 食材别名规范化
  -> 合并可见的默认基础调料
  -> 遍历当前活动 Recipe 集合
  -> 应用排除、耗时和不辣硬条件
  -> 计算已有 / 缺少 / 可选与覆盖率
  -> 稳定分组和排序
  -> 浏览器分组渲染
```

问答流保持独立。浏览器选择菜谱和一个问题后调用现有流式端点；停止生成只中断当前浏览器
请求，不影响搜索服务和菜谱详情。

## 错误处理

- 没有有效已有食材：前端阻止提交，API 返回 422；
- 存在无法识别的输入：保留可识别食材继续匹配，并返回安全提示；
- 三组均无结果：建议减少排除条件或关闭耗时限制；
- Ollama 不可用或流中断：保留详情并允许重试，不清空主结果；
- 用户停止生成：明确标记已停止，不把部分回答伪装成完整回答；
- 活动菜谱数据不可用：沿用 readiness 和 503，不降级到其他版本；
- 错误文案不得暴露密码、连接串、请求头、堆栈或敏感本地路径。

## 验收标准

自动化测试必须覆盖：

- 食材别名、重复项、空白输入和基础调料开关；
- 可选食材不计入缺失；
- 排除食材、最长耗时和不辣条件零违规；
- 三个分组边界、每组 5 条上限和稳定排序；
- 覆盖率及已有、缺少食材可复算；
- 原有六类自由文本查询和结构化菜谱对比保持兼容；
- 搜索和详情不调用 LLM，问答继续使用独立流式端点；
- 默认测试不连接真实 Neo4j/Ollama。

浏览器验收必须确认：

- 两种入口均能独立完成主流程；
- 三组结果和匹配解释正确展示；
- 详情正确区分已有、缺少、基础调料和可选食材；
- 快捷问题、自定义问题、停止、重试和复制均可操作；
- Ollama 不可用时食材匹配和详情仍可用；
- 窄屏无阻断操作的横向溢出，仅用键盘可完成主流程。

质量门继续执行完整 pytest、Ruff、pip check 和 git diff check。真实环境回归不得降低 Phase 2
基线：固定 50 条查询 Recall@5 不低于 0.90，可靠硬条件违规为 0，预热搜索 P95 不超过
1000ms，预热回答首字 P95 不超过 3000ms。外部服务不可用时只能标记为“实现完成、真实
验收待执行”，不能宣称 Phase 3 交付完成。
